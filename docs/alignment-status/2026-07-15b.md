## Subagent 异步协议

映射出正式版的完整 Agent 派发协议（探针，非源码猜测）：

- **默认异步**：`forceAsync = isForkSubagentEnabled()`（`AgentTool.tsx:557`）在默认环境为真，所以 general-purpose 子代理走后台。
- **Task 立即返回** `"Async agent launched successfully.\nagentId: <id> ...\nThe agent is working in the background."`（`AgentTool.tsx:1328`），子代理后台并发运行。
- print 模式主循环**不等子代理**：收到 launched 后用下一个 end_turn 结束（`num_turns=1`），子代理在后台跑完（marker 文件生成），完成时主循环会收到一条 `[SYSTEM NOTIFICATION - NOT USER INPUT]` user 消息重入。
- **`run_in_background: false` 强制同步**（`shouldRunAsync` 条件，`AgentTool.tsx:567`）：tool_result 是子代理结果本身，主代理阻塞到子代理完成——这与 hare 原有实现一致。

### hare 改动

- `agent_tool.py`：`run_in_background=true` 时用 `asyncio.ensure_future` 后台跑子代理引擎，Task 立即返回 launched 消息；`false`（及缺省的显式同步 case）保持阻塞路径。此前 `run_in_background` 被读取但**从不影响控制流**——无论如何都同步执行。
- 同步 golden `subagent.task_dispatch_sync`（fixture 显式 `run_in_background: false`）锁住阻塞路径，与正式版一致（result=主代理话、num_turns=2、usage=160、子代理文件在）。
- 异步路径由单元测试 `test_hare_agent_async_dispatch.py` 锁住（返回 launched、不阻塞）。

### 为什么异步没有 golden

异步子代理与主代理**并发地从同一 mock server 取响应**，响应流不再是确定的线性序列，位置式 fixture harness 无法可靠录制（录制会挂起）。这是 harness 的结构性限制，不是 hare 的 bug。异步行为改用单元测试覆盖。

## 退化空文本 fixture

`chat.empty_text` / `chat.whitespace_result` 用空 / 纯空白模型响应。**正式版把这类响应判为无效并重试 7 次直到挂起**（探针确认，非偶发）。这两个 case 对齐的是一个正式版都不接受的退化输入，已删除（含 fixture 和 golden）。case 数 64 → 62。
