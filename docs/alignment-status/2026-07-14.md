## TS oracle 挂起：根因已定位（当日主要成果）

通过隔离复现矩阵（mock server + 请求计数 + 变体对照）确认了三个事实：

1. **挂起触发条件是"项目里存在 `.claude/settings.json`"本身**，连空 `{}` 都会触发；挂起发生在启动阶段（mock server 收到 0 个请求），与 Bash 执行无关。根因是**工作区信任检查**：未信任的工作区遇到项目 settings 会走 trust dialog，recovered 构建在 headless 下渲染不了对话框就永久等待。
2. **recovered 构建（`bun src/entrypoints/cli.tsx`）的 headless print 模式本身已不可用**：即使无 settings、预置信任，也是静默 exit 0、零 API 调用、零输出（这与 `record_golden.py` 里"emits no JSON result"注释和 `session.resume_bad_id` 的空输出分歧互相印证）。**它不能再作为录制 oracle。**
3. **已提交的富输出 golden（如 `permission.deny_bash`）实际来自真实安装的 `claude` CLI**：用本机 `claude` 2.1.207 重录 deny_bash，结果与已提交 golden 一致（仅计时值和新版新增的 `ttft_stream_ms` 字段有差）。用 2.1.207 + 预置信任跑 settings-allow 场景：1.9s 干净退出、2 次 API 调用、tool_result 为 `"marker-42"`——**allow 规则真实生效，permission 轴 oracle 完全可用**。

### 已落地的修复

- `scripts/record_golden.py` 与 `hare/scripts/record_golden.py`：录制前在临时 `CLAUDE_CONFIG_DIR/.claude.json` 预置 `hasCompletedOnboarding` + `projects[<sandbox>].hasTrustDialogAccepted: true`，使项目 settings 在 print 模式被真实采纳、录制不再阻塞。

## 第二轮：permission / MCP / hooks 三轴推进（同日）

oracle 打通后一次性补了 6 个 golden，**每一个都抓到了真实 bug**。E2E 从 69 涨到 75 passed。

### 新增 golden（6 个，均由 2.1.87 oracle 录制）

| case | 覆盖 |
|---|---|
| `permission.settings_deny_read` | settings deny 规则拦截 Read |
| `permission.mode_bypass` | `--permission-mode bypassPermissions` |
| `permission.settings_precedence` | settings.json allow + settings.local.json deny → deny 赢 |
| `permission.settings_allow_redirect_denied` | `Bash(echo *)` allow 规则**不**覆盖 `echo x > file` |
| `mcp.stdio_tool_call` | MCP stdio 工具真实调用 |
| `hooks.pretool_block` | settings 声明的 PreToolUse command hook 拦截 Bash |

**关键方法**：scripted fixture 下模型文本是写死的，allow 与 deny 的 stdout 完全相同——所以光比对 stdout 证明不了工具真的跑了。两个补强手段：
1. `_stable_result` 纳入 `permission_denials`（permission 轴唯一的结构性判别信号）。
2. 让被测组件在沙箱里落**标记文件**（MCP echo server 写 `mcp_echo_called.txt`、hook 写 `hook_pretool_ran.txt`），用 `check_files` 做真实执行证据。MCP 和 hooks 两个 bug 都是靠这个抓出来的——它们的 stdout 一直是"对"的。

### 抓到并修复的真实 bug（4 类）

1. **输出重定向从不校验**（`hare/tools_impl/BashTool/`）：`Bash(echo *)` allow 规则让 `echo x > file` 直接写盘。移植 `validateOutputRedirections` + `isPathAllowed` 的 create 门：重定向目标即使在 cwd 内也需要写权限。
2. **`ask` 决策不阻断工具**（`hare/query/core.py`、`hare/services/tools/tool_execution.py`）：两个执行器都只在 `deny` 时阻断，`ask` 会照常执行。`toolExecution.ts:995` 是对任何非 allow 决策都阻断。
3. **MCP 工具从未真正执行**（`hare/services/mcp/runtime.py`）：`_McpRuntimeTool.call` 签名是 `(args, **kwargs)`，而执行器按位置传 `(input, context, can_use_tool, assistant)` → 每次调用抛 TypeError，被吞成工具错误。07-13 声称的"MCP 接线完成"其实是假的。
4. **hooks 管线是死代码**（`hare/services/tools/`、`hare/utils/hooks/`）：`run_pre_tool_use_hooks` 没有任何调用方；settings 里的 hooks 从未注册（`hooks_settings.get_all_hooks` 读的是不匹配的扁平 schema 且无运行时调用方）；command hook 也没拿到 stdin payload。三处全部补齐，新增 `hare/utils/hooks/settings_hooks.py`。

顺带：`permission_denials` 曾把 `passthrough` 误记为 denial——TS 的 canUseTool 只返回 allow/deny/ask，passthrough 是 hare 的"无规则匹配"终态，headless 下等同放行，不算 denial。

### 参考行为记录（供后续 case 参考）

- hook 拦截**不**计入 `permission_denials`（与规则 deny 不同）。
- 工具级 deny（`"deny": ["Bash"]`）参考实现是把工具**移出工具表**（`No such tool available: Bash`），不是运行时拒绝。
- 无规则的普通命令在 print 模式会被参考实现放行（走 auto-mode classifier）；hare 用 passthrough 达到同样效果。

### compact 轴：从"静默失败"到对齐

`compact_messages` / `strip_images_from_messages` 假设消息是 dict，遇到 loop 的消息对象直接抛 AttributeError，被 `auto_compact_if_needed` 吞成失败计数——**auto-compact 从来没成功运行过**。除崩溃外还有 3 个真实差距：

`subagent.task_dispatch`：子代理跑起来了、文件副作用也对，但 **hare 的主引擎没有隔离子代理**——主引擎的 usage 总量把子代理的轮次算了进去（190 vs 参考的 160），Task 返回后主循环不再走一轮，最终 result 是子代理的最后一句而非主代理自己的。golden 已录（参考真相在案），留给下一轮修。

### harness 修复（影响所有多次调用 case）

hare 每次调用起一个新进程、fixture 从响应 0 重放；而参考把**所有**调用打到同一个 mock server、响应流持续推进。两侧语义根本不等价（`session.resume_basic` 只是因为两个响应雷同才侥幸通过）。已加跨进程游标，且**每次调用都重读**——子代理引擎和父引擎因此共享同一条响应流。
