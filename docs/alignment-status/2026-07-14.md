## TS oracle 挂起：根因已定位（当日主要成果）

通过隔离复现矩阵（mock server + 请求计数 + 变体对照）确认了三个事实：

1. **挂起触发条件是"项目里存在 `.claude/settings.json`"本身**，连空 `{}` 都会触发；挂起发生在启动阶段（mock server 收到 0 个请求），与 Bash 执行无关。根因是**工作区信任检查**：未信任的工作区遇到项目 settings 会走 trust dialog，recovered 构建在 headless 下渲染不了对话框就永久等待。
2. **recovered 构建（`bun src/entrypoints/cli.tsx`）的 headless print 模式本身已不可用**：即使无 settings、预置信任，也是静默 exit 0、零 API 调用、零输出（这与 `record_golden.py` 里"emits no JSON result"注释和 `session.resume_bad_id` 的空输出分歧互相印证）。**它不能再作为录制 oracle。**
3. **已提交的富输出 golden（如 `permission.deny_bash`）实际来自真实安装的 `claude` CLI**：用本机 `claude` 2.1.207 重录 deny_bash，结果与已提交 golden 一致（仅计时值和新版新增的 `ttft_stream_ms` 字段有差）。用 2.1.207 + 预置信任跑 settings-allow 场景：1.9s 干净退出、2 次 API 调用、tool_result 为 `"marker-42"`——**allow 规则真实生效，permission 轴 oracle 完全可用**。

### 已落地的修复

- `scripts/record_golden.py` 与 `hare/scripts/record_golden.py`：录制前在临时 `CLAUDE_CONFIG_DIR/.claude.json` 预置 `hasCompletedOnboarding` + `projects[<sandbox>].hasTrustDialogAccepted: true`，使项目 settings 在 print 模式被真实采纳、录制不再阻塞。
- 差分 case 的 seed 约定：TS 侧读 `.claude/settings.json`，hare 侧读 `.hare/settings.json`（`hare/utils/settings/settings.py:167`），permission×settings case 需要同时 seed 两个文件。

- npm 上无 2.1.88（recovered 快照版本），最近邻 `@anthropic-ai/claude-code@2.1.87` 已安装到仓库内 `.oracle/claude-code-2.1.87/`（已加入 `.gitignore`）。
