# Alignment Status - 2026-07-15 (c): content-matched fixture harness

## 目标与成果

位置式 fixture（`kind: scripted`）按调用顺序发响应。这对并发/异步流失效——父代理和子代理从同一 fixture 取响应、请求交错，第 N 个响应不再对应第 N 次调用。

新增 **`kind: content-matched`**：按请求内容选响应，而非顺序。这是解锁一整类并发/分支场景录制的基础设施。

### 实现

- `scripts/fixture_matching.py`（新）：共享匹配器。每个响应带 `match` 谓词（`last_tool_use` / `user_contains` / `no_tool_result` / `has_tool_result`，present 的全部 AND），可选 `once: true` 表示只消费一次。无 `match` 的响应是 wildcard fallback。
- mock server（TS 侧）与 hare fake_model（Layer A）都 import 同一个匹配器，两侧对同一 fixture 匹配一致。
- `_normalize_message` 兼容 API dict（TS 侧）和 hare message 对象（Layer A 侧），谓词逻辑不分叉。
- 匹配器 7 个单元测试（`tests/test_fixture_matching.py`），含对象归一化、once 消费、内容优先于顺序。

### 端到端验证

新 case `chat.content_matched_flow`（Read→Bash 的内容分支多轮流）在**两侧都跑通**：TS 侧用 content-matched 录制成功（`inspected.txt` 生成、num_turns=3），hare Layer A 对齐通过。证明基础设施端到端可用。

## 异步子代理：基础设施到位，但完整实现仍受阻

content-matched harness **成功录制了异步子代理的 TS golden**（`result: "Launched the agent in the background."`、num_turns=1、子代理后台建了文件）——这一步以前是挂起的，现在可行了。

但 hare 侧对齐仍失败，根因是**更深的架构缺陷**：hare 的异步分支用 `asyncio.ensure_future` 起后台子代理 task，而 print 模式是单次 `asyncio.run()`——主协程返回后 event loop 关闭，pending 的子代理 task 被直接丢弃，从未真正执行（num_turns=2 而非 1，子代理文件不生成）。

上一轮的异步分支用 `asyncio.ensure_future` fire-and-forget——在 print 单次 `asyncio.run()` 里这个 task 会被丢弃，子代理从不执行（比原同步实现更糟）。本轮把它改成**在工具内 drain 到完成**：子代理真正跑完、文件效果落盘，返回参考的 launched 消息；只把"父代理不等子代理"的并发性推迟。在 print 单次模式下父代理反正立即结束，等或不等对最终结果无差别。异步单元测试（返回 launched、同步不含 launched）不变。

完整的并发异步（父代理拿到 launched 立即结束、子代理真后台跑、完成时 `[SYSTEM NOTIFICATION]` 重入）需要 hare 主循环支持跨轮后台任务生命周期，作为后续任务。因此本轮**不保留**无法对齐的 async golden，只保留 content-matched 基础设施 + 一个能证明它工作的 case。
