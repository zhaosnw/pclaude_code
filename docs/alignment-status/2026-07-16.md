# Alignment Status - 2026-07-16: async subagent lifecycle

承接 content-matched harness（07-15c）——它解锁了异步录制，但当时 hare 的异步分支只是 drain-to-completion（子代理跑完但没有真正的并发 + 完成重入）。本轮实现完整的异步子代理生命周期。

## 协议（从 2.1.209 探针捕获，非源码逆向）

recovered 源码里 `registerAsyncAgentTask` / `notifyOnCompletion` 被 mangle，只剩注释。改用探针捕获正式版的完整行为：

1. **默认异步** + `run_in_background: false` 强制同步（已在 07-15b 确立）。
2. Task 立即返回 `"Async agent launched successfully.\nagentId: ..."`，子代理**后台并发**运行。
3. 子代理完成后，主循环收到一条 user 消息重入，格式是 `[SYSTEM NOTIFICATION - NOT USER INPUT]` + `<task-notification>` 块，字段：`task-id` / `tool-use-id` / `output-file` / `status` / `summary`（`Agent "<desc>" finished`）/ `note` / `result`（子代理最后文本）/ `usage`（subagent_tokens / tool_uses / duration_ms）。原文逐字捕获。
4. 主代理据此给出最终回复。`num_turns: 1`——重入不新增 turn。

## hare 实现

- **`hare/tools_impl/AgentTool/async_agent_tasks.py`**（新）：进程级后台任务注册表 + 完成队列 + `build_task_notification`（渲染探针捕获的重入原文）。
- **Agent 工具异步分支**：`run_in_background=true` 用 `asyncio.ensure_future` 起后台子代理 runner（真并发，不再 drain），记录完成（result / tokens / tool_uses / duration），返回 launched。
- **QueryEngine 重入**：主 query 循环结束后，`while has_pending()` 排空完成队列，把每个完成作为 `<task-notification>` user 消息注入，再跑一轮 query 处理重入，assistant 回复 yield 给主流程。**子代理引擎 gate**：`is_subagent`（config.agent_id 非空）时不排空——否则子代理会等父代理拥有的任务，死锁。

### 端到端结果（golden `subagent.async_dispatch`，content-matched fixture）

- ✅ launched 返回、子代理后台跑、文件效果 `sub_async_ran.txt` 落盘
- ✅ 完成重入触发、主代理最终 result `"The background agent finished."` 与参考一致
- ⚠️ `num_turns`：hare 2 vs 参考 1 —— 登记为 `known_divergence`

### num_turns 差异（已登记）

参考不把异步 launch 的 tool_result 计为 turn，hare 把每个 tool_result 计为 user turn。干净修复需要改 turn 计数语义（触及所有 case），风险不匹配一个计数字段的收益；用 launched-message 内容 heuristic 又太脆弱。故登记 `known_divergence` 而非 hack 掩盖。**异步生命周期的所有实质行为均已对齐**，仅此计数字段差 1。
