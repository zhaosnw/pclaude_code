## 当前已确认状态

### E2E 与路径

- E2E 当前以 `hare/alignment/` 作为主目录。
- 真实 case 的 fixture 引用当前已经统一到 `hare/alignment/fixtures/...`。
- root / `hare/` 两份 `scripts/e2e_runner.py` 与 `scripts/record_golden.py` 当前都兼容：
  - `hare/alignment/fixtures/...`
  - `alignment/fixtures/...`
- transcript 路径当前使用 project-scoped 逻辑，并支持 `HARE_CONFIG_DIR` / `CLAUDE_CONFIG_DIR`。
- sandbox 相关 subprocess 测试当前已使用隔离配置目录，不再把 `~/.hare/` 当默认写入目标。
- 主线 `tests/` / `hare/tests/` 之间的重复 basename 当前已经清零；剩余“重复问题”主要是目录层面的资产与代码树并存。

### 守卫与边界

- 旧 fixture 前缀与 root `alignment/` 直接引用，当前都有 allowlist 守卫测试。
- `alignment-guardrails` 已经同时存在于 root 和 `hare/` 子项目入口。
- 当前 canonical 与兼容层的边界已经明确，但还没有从结构上彻底去掉重复资产。

## 当前主要问题

### 1. 目录级资产重复仍未彻底收口

- 仓库当前仍并存 `hare/alignment/`、`legacy_alignment/`、root `alignment/` 三层资产目录。
- root / `hare/` 双层代码树仍会持续制造路径歧义、重复维护和统计放大。

### 2. `mypy` 质量门仍失效

- 当前基线仍是 `210`，实测结果是 `1011 errors`。
- 这意味着现有门禁不能真实反映代码健康度，也不能有效阻止新回归。

### 3. 真差分覆盖仍然偏薄

- 当前通过的重点还是主链路 E2E，不等于更大范围行为已经完成对齐。
- 仍值得继续补齐的轴包括：
  - `--resume` / `--continue`
  - subagent / Agent 嵌套执行
  - hooks
  - MCP
  - permission modes × settings
  - compact / auto-compact
  - transcript JSONL schema

## 当前建议

1. 继续收口 root `alignment/` 的活跃依赖面，避免新增入口默认回到兼容层。
2. 持续用守卫测试固化“当前 canonical 目录”和“兼容保留层”的边界。
3. 重新定义 `mypy` 基线，或者直接按模块启动清偿。
4. 后续优先补真差分覆盖，不再继续扩张重复资产面。
