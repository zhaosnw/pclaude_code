## 0. 先处理一个紧急问题：版本控制状态

评审中发现的最高优先级事项，与代码质量无关，但风险最大：

- 工作区有 **2027 个未入库变更**：1315 个删除（HEAD 里残留的旧扁平布局 `hare/*.py` 的过期副本）+ **708 个未跟踪文件**。
- 未跟踪文件里有 **463 个是 `hare/hare/` 下的 Python 源码**、77 个测试文件、20 个脚本——**约三分之一的代码库从未被 git 保护过**。一次误删 / 磁盘故障 / `git clean` 就会永久丢失。
- 历史原因：项目曾从 `<repo>/hare/*.py` 重组为嵌套的 `<repo>/hare/hare/*.py`（hare/ 变成完整项目目录，含 Makefile/docs/tests），但这次重组从未整体提交，后续提交只 add 了当次改动的文件。

**建议立即执行**（一次提交解决）：

```bash
git add -A hare/ && git rm -r --cached $(git ls-files 'hare/*.py' | grep -v '^hare/hare/') 2>/dev/null
# 或者最简单：git add -A 后审一眼 git status 再 commit
git commit -m "chore: commit the hare/ nested-project restructure in full"
```

注意顺手把 `hare/.coverage`、`hare/coverage.xml`、`__pycache__`、`hare/frontend/node_modules` 等加进 `.gitignore` 而不是入库。

### 1.3 "完成"的定义需要你拍板

当前事实上的范围决策（散落在代码注释里，没有一份明确文档）：

1. **headless/print 模式是第一公民**——`-p`、`--output-format json/stream-json`、resume/continue 这条链路是对齐重点，也是 E2E 差分的验证对象。
2. **交互式 REPL 是"简化版"**（`main.py:623` 自述 "Simplified version of the React/Ink REPL"），593 个 UI 文件、12.7 万行 TUI 代码不在移植范围。
3. bridge（远程会话编排）文件在但深度浅。

**建议**：写一份 `SCOPE.md` 把"哪些子系统追求行为等价、哪些是简化替代、哪些明确不做"定下来。没有这份文档，"是否完成"永远没有可判定的答案。

### 1.4 表面完成度 vs 真实完成度

- 表面很干净：全库仅 **6 个 `NotImplementedError` + 7 个 TODO**（`make stub-check` 通过，P0/P1 模块零 stub）。
- 但 stub 计数器只能测"标记"，测不出"写了但写浅了"。真实深度要靠差分测试暴露——而最近的差分审计**每轮都还能挖出 confirmed bug**（见 §2.3），这是"未完成"最诚实的信号。

## 2. 正确性验证评估

### 2.1 现有验证体系（设计是好的）

五层金字塔，全部实跑确认绿灯（2026-07-02）：

| 层 | 内容 | 本次实跑结果 |
|---|---|---|
| 单元测试 | `tests/`，无网络无子进程 | **1,144 passed**（5.9s） |
| 属性测试 | Hypothesis：message/path/permission 不变量 | 含在上行 |
| alignment 套件 | 519 个函数级 case（root `alignment/cases/`，进程内 mock 跑） | **1,585 passed, 1 skipped**（131s） |
| E2E golden 差分 | 41 个 case：hare 走 fixture 注入，golden 从**真 TS CLI** 录制；含文件副作用断言、请求侧断言、工具注册表 parity | **64 passed**（9.7s） |
| live 冒烟 | 真模型压测（`HARE_LIVE_TESTS=1`，计费） | 未跑（历史记录见 docs/alignment-findings.md） |

E2E 差分的架构（`hare/docs/e2e-testing.md`）是这个项目最有价值的资产：同一份 fixture，hare 走进程内注入（Layer A），TS 原版走 mock SSE server（Layer B），同一 normalizer 后逐字节比对。`known_divergence` 机制（真实差异标 xfail 而不是改 golden 抹掉）也是正确的做法。

### 2.2 验证体系的四个薄弱点

**① 真差分覆盖太薄。** 拿 TS 原版当 oracle 的只有 41 个 golden case（38 个 TS 录制 + 3 个 self）。519 个 alignment case 是 **`--py-only` 自比对**（Makefile 里 `--ts /dev/null`）——它是好的回归护栏，但 expected 值是开发时写的，写错了就一直"绿"。5 月 22 日跑过唯一一次 50-case 的真 TS↔PY 对比（`alignment-artifacts/`，全过），至今 40 天没再跑过。

**② oracle 版本漂移。** golden 用本机官方 `claude` **2.1.165** 录制，hare 复现的是 **2.1.88**。两版之间的行为差异会被误判为 hare 的 bug（或反过来掩盖真 bug）。恢复出来的 2.1.88 TS CLI headless 模式卡死（文档已记录），这个问题不修，oracle 永远不纯。

**③ 收敛没有达成，且有量化证据。** 最近 4 个提交：`0e8c226`（resume 读路径整个缺失）、`3b37f72`（5 个 confirmed bug）、`7cebdbf`（2 个）、`09cc16d`（4 个，含 auto-compact 拿 200k 硬编码当 1M 模型窗口这种高危错误）。**每一轮对抗性审计都能挖出真 bug** = 当前测试全绿只说明"测过的都对"，覆盖外仍是雷区。

**④ 静态质量在回归。** mypy 当前 **497 errors / 174 files**，Makefile 基线是 210——翻了一倍多，`make mypy-regression` 这个门已经形同虚设。coverage.xml 是 5 月 31 日的，早已失真。

### 2.3 验证路线图建议（按优先级）

1. **P0 — 扩真差分的"轴"而不是"量"**。41 个 case 集中在 chat/tools/json 输出上。缺的轴（每轴先 3-5 个 case）：
   - `--resume`/`--continue`（刚修完大 bug 的区域，最该锁住）
   - subagent/Agent 工具嵌套执行（已有 xfail 起点）
   - hooks（PreToolUse/PostToolUse/Stop 的执行与拦截）
   - permission modes × settings 优先级矩阵
   - MCP（stdio server 的 list/call/resources）
   - compact/auto-compact 触发边界（09cc16d 刚修过阈值语义）
   - 会话持久化 JSONL 的字节级 schema 对比
2. **P0 — 把真差分放进 CI 定期跑**（哪怕 nightly）：`record_golden.py` 依赖本机 `claude`，把 TS 版本号写进 golden 元数据，版本升级时强制重录并甄别漂移。
3. **P1 — 请求侧差分常态化**。输出对齐骗得过 fixture，骗不过请求体。已有 `test_request_side_alignment.py` 的思路很对：扩展到 system prompt 全文、tools schema 全量、context_management 等字段的快照对比。
4. **P1 — 修恢复版 2.1.88 TS CLI 的 headless 卡死**，让 oracle 回到同版本。修好前，所有 2.1.165 golden 标注版本来源。
5. **P1 — 治理 mypy**：要么把基线升到 497 并冻结（诚实），要么分模块清偿。P0 模块（query/tools/services.api）建议清零。
6. **P2 — 把 519 个 py-only case 逐步"升级"为真差分**：alignment_runner 已有跑 TS 侧的能力（5 月 22 日用过），挑 P0 优先级的 case 补 TS 对照。
7. **P2 — 属性测试扩面**：现在只有 3 个文件（message/path/permission）。permission 规则解析、shell 词法（`excludedCommands ':*'` 这种刚出过 bug 的地方）、JSONL roundtrip 都是 Hypothesis 的好靶子。

## 3. 架构评审

### 3.1 做对了的（应当保持）

1. **镜像移植 + 溯源头**。984 个文件带 `Port of: src/xxx.ts` 注释，部分精确到行号（`deps.py` 里 `-- src/query/deps.ts L21-31`）。这让"对着 TS 逐函数审"成为可能——最近几轮 adversarial audit 能高效挖 bug，靠的就是这个。
2. **依赖注入缝（`query/deps.py`）**。`QueryDeps` 把 call_model/compact/uuid 收口，`HARE_MODEL_FIXTURE` 环境变量让真 CLI 子进程可以整机确定性回放——这是整个 E2E 差分体系的地基，设计干净。
3. **语义对等的翻译策略**。TS async generator → Python async generator，状态机形状保持一致（`query/core.py` 自述），没有强行"Python 化"重构。对以"行为等价"为目标的移植，这是正确选择；等对齐验收后再做 idiomatic 重构不迟。
4. **测试金字塔 + Makefile 工程化**。分层 marker、stub 门、对齐门、安全门（bandit）、覆盖率门齐全，`make all` 即 CI。

### 3.2 有问题的（按严重度）

1. **【高】版本控制卫生**——见 §0。463 个未跟踪源文件 + HEAD 里 1315 个过期旧路径文件。这不是风格问题，是数据风险。
2. **【高】两套 alignment 体系并存且语义混淆**。root `alignment/`（519 个 py-only 函数级 case，`scripts/alignment_runner.py`）和 `hare/alignment/`（41 个真差分 golden E2E case）名字相同、目的不同。Makefile 里 `alignment-quick` 实际是自比对，名字却暗示差分。建议：root 那套改名为 `regression/` 或明确标注 "py-only oracle"，文档里写清两者关系。
3. **【中】仓库里有三份 TS 源码副本**：`recovered-from-cli-js-map/`（含 node_modules！）、`hare/frontend/`（又一份 2.1.88-recovered，也含 node_modules）、外加 `claude-code-book/` 等。建议保留一份 vendor 化的参照（不带 node_modules），其余删除或子模块化。
4. **【中】根目录文档腐烂**。5 个大写报告（FINAL/COMPREHENSIVE/CHECKLIST/…）全部停在 5 月，数据（3.8/10、95k 行）与现状（208k 行）差一倍，而最新的真实状态藏在 `hare/docs/alignment-findings.md` 和 `hare/docs/e2e-testing.md` 里。建议根目录报告移进 `docs/archive/`，README 指向 hare/docs。
5. **【低】`utils/` 645 个文件**是 TS 侧结构的忠实镜像（TS 也是 564 个），移植期可接受；但它是 mypy 错误的主要来源地之一，验收后值得按域拆分。
6. **【低】简化 REPL 的边界未声明**。`ink_facade.py` + `_run_repl` 替代 React/Ink，功能差距（对话内 slash 命令、补全、快捷键、对话框等）没有清单。如果 REPL 属于交付范围，需要一份差距清单；如果不属于，写进 SCOPE.md。
