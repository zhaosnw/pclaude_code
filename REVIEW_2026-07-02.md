## 当前主要问题

### 1. 目录级资产重复仍未彻底收口

- 仓库当前仍并存 `hare/alignment/` 与 `legacy_alignment/` 两层对齐资产目录。
- 当前不再把 `hare/` 子项目目录视角当作受支持执行入口；根目录是唯一执行入口。

### 2. `mypy` 回归门已恢复，但类型债仍重

- 当前 canonical 扫描范围已排除 `hare/tests/`、前端、文档和脚本目录；`hare/hare/` 已被完全移除。
- 当前基线已更新为 `520`，`make mypy-regression` 与 `make -C hare mypy-regression` 都能通过并阻止新增回归。
- `make typecheck-full` 仍会展示完整的 `520` 个既有类型错误，后续需要按模块继续清偿。

### 3. 真差分覆盖（各轴现状）

Oracle 已钉定为 `.oracle/claude-code-2.1.87/`（npm 上无 2.1.88；recovered 构建的 headless print 模式已不可用，不能作为 oracle）。原先"录制挂起"的系统性阻塞已解决——根因是工作区信任检查。

各轴现状（详见 `docs/alignment-status/parity-matrix.md`）：

| 轴 | 状态 |
|---|---|
| `--resume` / `--continue` | ✅ 5 个 golden（含 resume-then-edit） |

| hooks | ✅ 7 个 golden，**10 个 P1 事件全部对齐** |
| MCP | ✅ 2 个 golden（stdio tool call 有真实调用证据） |

**关键方法论**：scripted fixture 下模型文本是写死的，工具跑没跑 stdout 都一样。所以 golden 必须带**真实执行证据**——`permission_denials` 字段、或被测组件在沙箱里落的 marker 文件。当日有 5 个"看起来早就完成"的功能（MCP、hooks、compact、subagent、resume-edit）被证明**根本不工作**，它们的 stdout 一直是"对"的。

1. 持续用 guardrails 防止 `hare/hare/` 镜像树回流，避免 runtime / tooling / docs 再回到双份路径语义。
